スプリントのライフサイクル

グループ作成 → sprints INSERT（index_num=0, started_at=now, ends_at=now+sprint_duration_days）
スプリント期間中 → UNIQUE(sprint_id, sent_by) で通知は1人1回まで / posts が on_time | late を判定
スプリント終了（Cron） → 未投稿メンバーに posts INSERT（status='missed'）→ 次スプリント INSERT（index_num+1）

ビジネス制約

制約テーブル意味
UNIQUE(group_id, index_num) sprints グループ内でスプリント番号は一意
UNIQUE(sprint_id, sent_by) notifications 1スプリント1人1回の通知発行
UNIQUE(notification_id, user_id) posts 1通知につき1ユーザー1投稿
UNIQUE(post_id, user_id, reaction_type) reactions 同じリアクションは1回のみ（トグルで削除）
PRIMARY KEY(group_id, user_id) group_members グループ所属の一意性

iOS モデル対応

DB テーブルiOS モデル主なマッピング
users GitHubUser / RepositoryMember github_login ↔ .login / avatar_url ↔ .avatarURL
groups Repository repo_full_name ↔ .name
group_members Repository.members JOIN → [RepositoryMember]
sprints なし（サーバー内部管理） グループ作成時に自動生成
notifications RepositoryNotification message ↔ .comment / sent_at ↔ .createdAt
posts RepositoryActivity (Dashboard) / 将来 FeedPost (Feed) post_type ↔ .type / body ↔ .comment
reactions RepositoryReaction / FeedPost.reactions reaction_type = heart|check (Dashboard) / lgtm|... (Feed)
github_webhook_deliveries なし（サーバー内部） 他テーブルと FK なし。Webhook 冪等性のみ
